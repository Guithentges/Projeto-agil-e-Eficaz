Visão Geral da Intervenção

Este documento detalha a transformação arquitetural realizada no Projeto Gil e Eficaz para garantir a escalabilidade multi-empresa e a integridade financeira.

O foco principal foi a remoção de lógica sensível do frontend e a implementação de defesas robustas no banco de dados (Supabase/PostgreSQL), garantindo que cada transação seja auditável e segura.

01. Hardening e Segurança Crítica

Proteção Anti-Fraude e IDOR

Implementamos uma defesa multicamadas para impedir que usuários manipulem dados de outras empresas ou burlem o estoque.

1. Validação de Ownership (Propriedade)

A RPC agora percorre todo o carrinho antes de começar a venda. Ela verifica se cada ID enviado pertence ao ID da empresa do usuário logado. Se houver tentativa de vender um item de outra empresa, a transação morre antes de afetar o banco.

2. Atomicidade de Estoque

Substituímos o comando simples de subtração por uma operação condicional. A venda só é autorizada se o saldo resultante for >= 0. Usamos GET DIAGNOSTICS para confirmar que o banco realmente alterou as linhas; se o saldo for insuficiente, um erro é disparado.

SCRIPT 05: SEGURANÇA CRÍTICASQL
-- Bloquear troca de empresa no perfil
CREATE OR REPLACE FUNCTION public.bloquear_mudanca_empresa()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.id_empresa IS DISTINCT FROM NEW.id_empresa THEN
    RAISE EXCEPTION 'Alteração de empresa não permitida.';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- RPC de Venda Blindada (Exemplo de Validação de Saldo)
UPDATE public."Estoque"
SET quantidade = quantidade - v_qtd
WHERE id_produto = v_id 
  AND id_empresa = p_id_empresa 
  AND quantidade >= v_qtd;

GET DIAGNOSTICS v_rows = ROW_COUNT;
IF v_rows = 0 THEN RAISE EXCEPTION 'Estoque insuficiente.'; END IF;

02. Onboarding Automatizado (Trigger System)

Experiência de Usuário Sem Atrito

Anteriormente, o sistema dependia do frontend para criar registros mestre, o que causava erros de RLS e falhas de sincronia.

Criação em Cascata

Agora, no momento que o usuário confirma o e-mail/cadastro, o banco dispara o script 04. Ele cria a Empresa, vincula o Perfil, atribui o cargo de Admin e, por fim, o script 02 gera todas as categorias operacionais padrão (Aluguel, Funcionários, etc) automaticamente.

SCRIPT 04: AUTOMATED ONBOARDINGSQL
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  -- 1. Criar Empresa
  INSERT INTO public."Empresa" ("Nome")
  VALUES (COALESCE(NEW.raw_user_meta_data->>'empresa_nome', 'Nova Empresa'))
  RETURNING id INTO v_empresa_id;

  -- 2. Criar Profile e Vincular
  INSERT INTO public.profiles (id, nome, id_empresa)
  VALUES (NEW.id, NEW.raw_user_meta_data->>'nome', v_empresa_id);

  -- 3. Atribuir Cargo de Administrador
  INSERT INTO public.user_roles (user_id, role)
  VALUES (NEW.id, 'admin');

  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

03. Orquestração Segura (Edge Function)

O Porteiro Digital

A Edge Function `quick-task` é o único ponto de entrada para finalização de vendas, atuando como um firewall de aplicação.

Extração de Identidade

A função não confia no ID de empresa enviado pelo cliente. Ela extrai o sub (ID do usuário) do token JWT assinado e busca o ID da empresa no banco de dados usando privilégios de administrador (service role). Isso garante que um usuário nunca consiga fingir ser outra empresa.

EDGE FUNCTION: FINALIZAR_VENDA.TSTYPESCRIPT
import { createClient } from "https://esm.sh/@supabase/supabase-js@2.39.3";

Deno.serve(async (req) => {
  const authHeader = req.headers.get("Authorization");
  const supabaseClient = createClient(URL, ANON_KEY, { global: { headers: { Authorization: authHeader } } });
  
  // Extrai ID do usuário com segurança
  const { data: { user } } = await supabaseClient.auth.getUser();
  
  // Executa RPC com Service Role (Bypass de RLS controlado)
  const adminClient = createClient(URL, SERVICE_ROLE_KEY);
  const { data: empresa } = await adminClient.from("profiles").select("id_empresa").eq("id", user.id).single();

  const { data, error } = await adminClient.rpc("processar_venda_atomica", {
    p_id_empresa: empresa.id_empresa,
    p_carrinho: body.carrinho
  });

  if (error) return new Response(JSON.stringify({ error: error.message }), { status: 400 });
  return new Response(JSON.stringify({ success: true, data }), { status: 200 });
});

04. Ajustes de Interface e UX

Navegação e Controle de Acesso

  • AppLayout: Adicionado guardião que detecta se o onboarding falhou. Em vez de erro genérico, o usuário recebe orientações de suporte.
  • PDV.tsx: Refatorado para enviar o carrinho simplificado para a Edge Function, delegando toda a complexidade de cálculo de estoque ao banco.
  • useAuth: Implementado estado de carregamento atômico para evitar que o ID da empresa "pisque" ou desapareça durante a navegação.

Relatório de Segurança e Auditoria

Este documento descreve as vulnerabilidades identificadas e as estratégias de mitigação recomendadas para o fortalecimento do sistema de PDV e Gestão de Vendas.

1 & 2. Centralização da Lógica de Negócio no Backend

O Problema: O frontend (React) está recebendo a responsabilidade de calcular o saldo de estoque e decidir como os insumos de um combo devem ser debitados. O sistema está aceitando o valor calculado na máquina do cliente, o que permite que qualquer usuário intercepte a requisição e envie saldos falsos ou negativos.
A Solução: Transfira toda a autoridade matemática para o servidor. O frontend em React deve enviar apenas a "intenção" da ação (exemplo: requisição informando "Vender 1 Combo X"). O cálculo de quais ingredientes compõem esse prato e a subtração do estoque devem ocorrer exclusivamente em rotinas de backend, utilizando Supabase Edge Functions ou RPCs (Remote Procedure Calls) diretamente no PostgreSQL. Dessa forma, o servidor processa a regra de negócios proprietária de forma invisível e segura, imune a manipulações externas.

3. Prevenção de Injeção de Scripts (XSS)

O Problema: A interface utiliza dangerouslySetInnerHTML para renderizar gráficos. Como administradores inserem textos livres (como descrições de produtos), a falta de higienização cria um vetor para ataques de Cross-Site Scripting (XSS), permitindo que códigos maliciosos afetem outros gestores.
A Solução: Implemente uma camada dupla de proteção. Primeiro, faça a sanitização da saída de dados utilizando bibliotecas robustas como o DOMPurify antes de renderizar qualquer conteúdo via dangerouslySetInnerHTML, garantindo a remoção de tags <script> e atributos nocivos. Em paralelo, aplique validação rigorosa de entrada de dados no momento do cadastro utilizando esquemas como o Zod para garantir que caracteres HTML não autorizados sejam rejeitados.

4. Implementação de Row Level Security (RLS)

O Problema: O controle de acesso está sendo feito apenas ocultando botões na interface visual do React. Como não há validação no servidor, qualquer usuário autenticado pode requisitar os dados confidenciais diretamente à API.
A Solução: É imperativo habilitar e configurar as políticas de Row Level Security (RLS) diretamente nas tabelas do Supabase. Crie políticas baseadas em cargos (RBAC) que inspecionem o token do usuário (auth.uid()) e seu nível de permissão no momento de cada requisição. Assim, o próprio banco de dados negará operações não autorizadas de leitura ou escrita.

5. Blindagem da Cadeia de Suprimentos

O Problema: O projeto possui vulnerabilidades ativas em bibliotecas NPM de terceiros (incluindo severidade alta no ambiente Vite), o que facilita a execução de código indesejado e compromete a postura de segurança geral da aplicação.
A Solução: Execute auditorias regulares (npm audit) e utilize npm audit fix para resolver vulnerabilidades automáticas. Para dependências críticas, atualize manualmente as versões no package.json para os patches mais recentes. Recomenda-se o monitoramento contínuo para evitar a defasagem de segurança.
